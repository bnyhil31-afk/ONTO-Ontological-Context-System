HOW TO USE THIS DOCUMENT

Each item has:
  [ ] Status      — unchecked, in progress, or complete
  Priority        — CRITICAL, HIGH, MEDIUM, LOW
  Effort          — SMALL (hours), MEDIUM (days), LARGE (weeks)
  Stage           — which growth stage this belongs to
  Notes           — what done looks like

Work CRITICAL items before HIGH. HIGH before MEDIUM.
Never ship with an unchecked CRITICAL item.

SECTION 5 — PREDICTED ROADBLOCKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

These are the problems most projects discover too late.
They are documented here so they are discovered on purpose.

[ ] 5.01  CRITICAL / MEDIUM / Stage 1
          GDPR vs Permanent Memory conflict
          The problem: Principle VII requires permanent memory.
          GDPR requires right to erasure. These conflict.
          Done when: Architectural solution designed, documented,
          and reviewed by counsel. Personal data separable
          from audit events without breaking either principle.

[ ] 5.02  CRITICAL / MEDIUM / Stage 1
          Safety pattern false positives
          The problem: CRISIS and HARM patterns may trigger
          on legitimate inputs. A medical researcher typing
          about suicide prevention gets a crisis response.
          Done when: False positive rate measured.
          Context-aware detection implemented.
          Override mechanism for verified professional users.

[ ] 5.03  CRITICAL / MEDIUM / Stage 2
          First enterprise security questionnaire
          The problem: Enterprise procurement requires
          completion of a vendor security questionnaire
          — often 200+ questions. Unprepared teams fail here.
          Done when: Standard security questionnaire
          pre-completed and maintained. Updated with
          each security improvement.

[ ] 5.04  HIGH / MEDIUM / Stage 1
          Principle VIII vs user feedback conflict
          The problem: A user who wants to improve the system
          may suggest changing a principle. The hash blocks it.
          This needs a clear, respectful, documented response.
          Done when: Process for principle feedback documented.
          Community discussion mechanism exists.
          Distinction between principles and configuration clear.

[✅] 5.05  HIGH / SMALL / Stage 1
          Cross-platform line endings
          The problem: principles.txt hash will differ between
          Windows CRLF and Unix LF line endings.
          A Windows user who edits and saves the file
          will break verification even without changing content.
          Done when: .gitattributes enforces LF line endings.
          setup.sh normalizes line endings on install.
          Documented in troubleshooting guide.

[ ] 5.06  HIGH / MEDIUM / Stage 2
          First public security vulnerability report
          The problem: Someone will find a vulnerability
          and report it. If there is no process, the response
          will be chaotic and trust-damaging.
          Done when: SECURITY.md exists with responsible
          disclosure process. Contact method published.
          Response SLA committed to. Hall of fame for reporters.

[ ] 5.07  HIGH / MEDIUM / Stage 2
          The first hostile fork
          The problem: Someone will fork ONTO, remove the
          principles, and build something harmful with it.
          Done when: Preservation document published.
          Community notification process established.
          Response template prepared. Legal counsel briefed.

[ ] 5.08  HIGH / MEDIUM / Stage 2
          Competing with yourself
          The problem: The commercial version and the
          open source version must not cannibalize each other.
          Done when: Clear feature differentiation documented.
          Open version always functional and maintained.
          Commercial version adds enterprise features only.

[ ] 5.09  MEDIUM / SMALL / Stage 1
          Python version fragmentation
          The problem: Python 3.7, 3.9, 3.11, 3.12 all
          behave slightly differently. A user on an old
          Raspberry Pi may have Python 3.7. A developer
          may have Python 3.12. Both need to work.
          Done when: Test suite runs against all supported
          versions via GitHub Actions matrix.
          Minimum version enforced with clear error message.

[ ] 5.10  MEDIUM / SMALL / Stage 1
          Database corruption recovery
          The problem: If memory.db is corrupted —
          power loss during write, disk failure, etc —
          the system has no recovery path currently.
          Done when: Backup strategy documented.
          Recovery procedure tested.
          Corruption detected on boot with clear message.

[ ] 5.11  MEDIUM / MEDIUM / Stage 2
          Burnout and bus factor
          The problem: Currently one person knows this
          project completely. If that person is unavailable
          — illness, life events, burnout — the project stalls.
          Done when: At least one other person fully understands
          the project. Knowledge transfer documented.
          Preservation document covers succession.

[ ] 5.12  MEDIUM / MEDIUM / Stage 2
          AI regulatory changes
          The problem: AI regulation is moving fast.
          The EU AI Act, US executive orders, and others
          may impose requirements that affect ONTO.
          Done when: Legal counsel monitors regulatory
          changes. Quarterly review of compliance posture.
          Architecture flexible enough to adapt.

[ ] 5.13  LOW / MEDIUM / Stage 3
          The enterprise pilot that fails publicly
          The problem: The first enterprise pilot will
          not be perfect. If it fails publicly it damages
          trust significantly.
          Done when: Pilot program structure designed.
          Success criteria agreed in advance.
          Failure handling process documented.
          Communications plan prepared.

[ ] 5.14  LOW / SMALL / Stage 2
          Name collision
          The problem: ONTO may already be in use
          as a product name somewhere.
          Done when: Trademark search completed globally.
          Conflicts identified and resolved before launch.
