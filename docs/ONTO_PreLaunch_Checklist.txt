              inputs. Whether this constitutes personal data under
              GDPR Article 4(1) depends on whether individuals are
              identifiable from it. Single-user local deployments are
              lower risk. Multi-user deployments require formal review.

              Note: CRISIS response text exists in core/config.py.

  [ ] 5.09 — Mental health professional review of sensitive/crisis sets
              _SENSITIVE_CONCEPTS and _CRISIS_CONCEPTS in graph.py are
              intentionally conservative initial sets.
              Before deploying to any user population that may include
              vulnerable individuals, have a qualified mental health
              professional review and expand these sets.
              Note: crisis detection uses full-text substring matching,
              not just extracted concepts — more robust by design.

  [ ] 5.10 — Safe messaging compliance for crisis_detected pathway
              When relate() returns crisis_detected=True, verify the
              full pipeline (intake → checkpoint) surfaces:
                — Immediate crisis resources
                — Language compliant with AFSP/SAMHSA safe messaging
                — No storage of crisis content in graph
              Builds on existing item 5.08.

  [ ] 5.11 — Graph bias audit
              Co-occurrence graphs are known to encode and amplify
              bias (Bolukbasi et al. 2016 — Word2Vec gender bias).
              Personal graphs may be worse: they capture individual
              biased associations with no population-level averaging.
              Before any multi-user deployment:
                — Sample navigation results for systematic skew
                — Verify PPMI approximation does not amplify
                  commonly stereotyped associations
                — Consider minimum co-occurrence threshold before
                  an edge becomes traversable (require 2+ inputs)

  [ ] 5.12 — Graph entropy / echo chamber monitoring (design item)
              Design a graph health metric that detects when the graph
              becomes excessively narrow — a small number of hub
              concepts with disproportionate edge weight.
              Recommended threshold alert in navigate() when top-3
              results account for >80% of total effective weight.
              No code change required now — design document only.

SECTION 6 — STAGE 2 PREPARATION
(Design decisions to make now so Stage 2 upgrades require no
schema or interface changes. Not Stage 1 exit criteria.)

  [ ] 6.01 — Design directed edge schema (Stage 2 prep)
              Current: undirected edges (UUID-normalised direction).
              Stage 2: directed + typed edges (is-a, causes, part-of).
              Design the schema extension now so it is backward-
              compatible with existing undirected edges.

  [ ] 6.02 — Design Personalized PageRank interface (Stage 2 prep)
              PPR replaces BFS depth-2 traversal.
              Design: navigate() signature stays identical.
              Only _bfs_navigate() is replaced internally.
              Confirm NetworkX would be the Stage 2 dependency.

  [ ] 6.03 — Design full PPMI matrix approach (Stage 2 prep)
              Current Stage 1 approximation uses per-edge data only.
              Full PPMI requires: total concept count, per-concept
              frequency, per-pair co-occurrence count.
              The schema already stores what is needed. Document
              the exact PPMI formula to implement in Stage 2.

  [ ] 6.04 — Design per-user decay calibration (Stage 2 prep)
              Sense et al. (2016): forgetting rates vary between
              individuals but are stable within individuals.
              Design: DECAY_EXPONENT becomes a per-user parameter
              stored in a user_preferences table. Default 0.5.
              After 30 days of use, offer calibration based on
              which forgotten edges the user reintroduces.

  [ ] 6.05 — Evaluate spaCy for concept extraction (Stage 2 prep)
              Current RAKE-inspired extraction is stdlib-only.
              spaCy NER + noun phrase extraction would dramatically
              improve concept quality for technical and medical text.
              Evaluate: package size, mobile constraints, cold start.
              The _extract_concepts() contract is unchanged —
              just swap the implementation.
