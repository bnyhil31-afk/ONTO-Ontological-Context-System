PHILOSOPHY

This roadmap is not a schedule. It is a sequence.

Each stage is complete in itself. A deployment at Stage 1 is not
a degraded Stage 3 — it is a fully valid, self-sufficient system.
A node that never federates is not a lesser node. It is an organism
that chose to live independently, which is its right.

Stages unlock capability. They do not confer legitimacy.
The principles apply equally at every stage.

The sequence is designed so that every step forward is safe,
reversible in principle, and honest about what it adds and
what it requires.

STAGE 4 — P2P / PUBLIC COMMONS
Status: Not started
Target audience: Everyone. Any individual, any organization,
                 any scale, any jurisdiction.

What it adds over Stage 3:
  Fully peer-to-peer operation.
  No central infrastructure required.
  The protocol is public infrastructure — like HTTP.
  Discovery without a central registry.
  The network governs itself.

What a Stage 4 deployment looks like:
  Anyone can run a node with no permission required.
  Any node can find and connect to any other node.
  Context flows across organizational and jurisdictional boundaries.
  The network is resilient — no single point of failure.
  The values are enforced by the protocol, not by any authority.
  ONTO is infrastructure, not a product.

Key deliverables:
  ⬜ Distributed discovery protocol (no central registry)
  ⬜ Sybil resistance at scale
  ⬜ Post-quantum cryptography migration (Kyber-768)
  ⬜ Anti-concentration routing at network scale
  ⬜ Self-sovereign identity integration
  ⬜ Decentralized governance (DAO or cooperative model)
  ⬜ Carbon footprint routing (prefer greener paths)
  ⬜ Network health monitoring without central observer
  ⬜ Protocol commons legal entity established
  ⬜ Long-term sustainability model

Exit criteria for Stage 4:
  Network operates without any central infrastructure
  Governance functions without founders' direct involvement
  Protocol is adopted by entities the founders have no relationship with
  The values are preserved under conditions the founders did not design for

BEYOND STAGE 4 — FORWARD THINKING

These are not deliverables. They are directions.
They are included because forward-focused design means
knowing what you are pointing toward, even if you
cannot see the destination yet.

  Intergenerational continuity
    The system outlives its creators. Future generations can
    read, verify, and build on everything produced today.
    The audit trail is a historical record, not just a log.

  Ecological integration
    The system's computational carbon footprint is measurable,
    reportable, and actively minimized as a first-class property.
    The network prefers sustainable infrastructure.

  Universal accessibility
    The simplest possible deployment — a single command on the
    most modest hardware — is always maintained as a first-class
    experience. Complexity is hidden, not removed.

  Living governance
    The governance model evolves with the community it serves.
    It is never frozen. It is never captured. It is never
    more complex than it needs to be.

  Self-awareness
    The system monitors its own second and third order effects
    and reports them honestly. It knows what it is doing to
    the world and says so.

INPUT ADAPTER ARCHITECTURE — UNIVERSAL INPUT DESIGN

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ONTO is not a text system. It is a context system.

The three governing axes — distance, complexity, size — are
deliberately data-type agnostic. They describe any input,
regardless of format. The five-step loop does not care what
the raw input was. It cares about the package.

This means ONTO is designed to accept any input type that
can be converted into a classified, weighted intake package.

The architecture:

  Raw input (any format)
       ↓
  INPUT ADAPTER — converts format to intake package
       ↓
  intake package (standard dict — same for all types)
       ↓
  CONTEXTUALIZE → SURFACE → GOVERN → REMEMBER
  (unchanged — format-agnostic)

The intake package contract is the universal interface.
Every adapter produces the same structure. Everything
downstream is already built to handle it.

Current adapter (Stage 1):
  ✅ Text / string — intake.receive()
     Covers: questions, commands, statements, numbers,
             any human-typed input

Planned adapters by stage:

  Stage 1 (current sprint):
  ⬜ Structured data — JSON, CSV, database records
     The system can already classify and process structured
     inputs when passed as strings. A dedicated adapter adds
     field-level classification and relationship mapping.

  Stage 2 (intranet / small team):
  ⬜ Documents — PDF, DOCX, Markdown
     Pre-process to extract text, metadata, and structure.
     Preserve provenance (author, date, source).
  ⬜ System events — logs, alerts, API calls, webhooks
     Ingest from external systems. Classify by event severity.
     Treat each event as a first-class context input.
  ⬜ Other AI outputs — treat model outputs as inputs
     The output of one AI system becomes context for ONTO.
     Source type: derived. Provenance discount applied.

  Stage 3 (federated / enterprise):
  ⬜ Images — with appropriate vision processing module
     Classification at intake based on detected content.
     Shell stores metadata only — payload encrypted.
  ⬜ Audio — speech and environmental sound
     Transcription as pre-processing step.
     Original audio stored at classification level of content.
  ⬜ Sensor data — IoT, biometric, environmental streams
     Source type: sensor. Continuous ingestion support.
     Temporal decay applied — older readings weight less.

  Stage 4 (P2P / public commons):
  ⬜ Network signals — context from other ONTO nodes
     The most important adapter. Context arrives from peers.
     Provenance verification is mandatory.
     Source type: derived. Trust level from node identity.
  ⬜ Video — frame-level and temporal context
     High complexity, high size. Governed accordingly.

What every adapter must do:
  1. Convert raw input to a classified intake package
  2. Set source_type correctly (human | system | sensor | derived)
  3. Apply classification based on content, not format
  4. Preserve provenance — where did this come from?
  5. Never introduce synthetic data — every edge traces to input

What no adapter may do:
  - Bypass the safety check layer
  - Bypass classification
  - Auto-proceed past GOVERN without human authorization
  - Strip provenance information

The human sovereignty checkpoint applies to all input types.
A sensor reading that crosses a consequence threshold
requires the same human review as a text input that does.
The format changes. The principles do not.
